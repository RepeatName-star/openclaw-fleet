# v0.1 Scope (Feature Checklist + Boundaries)

This document is the v0.1 contract: what is in-scope, what is explicitly out-of-scope, and what is deferred to roadmap.

It complements `docs/bulk-management.md` (semantics) with a feature checklist across the original 6 requirements.

Implementation plan: `docs/plans/2026-03-11-bulk-management-v0.1.md`.

## In-Scope (v0.1)

### 1) Group/Tag Based Bulk Management

- Labels
  - Business labels: operator-managed (writeable), enforced to K8s label syntax.
  - System labels: `openclaw.io/*` (read-only, highest priority).
  - Conflict rule: business writes that conflict with system labels are rejected.
- Selector / Group
  - `Group = named Selector` (label selector only, K8s label selector semantics).
  - Dynamic membership (instances enter/leave scope over time).
- Campaign (fan-out)
  - First-class object for batch execution (fan-out to every matching instance).
  - Best-effort reconciliation: run eligible first, blocked remain visible, auto-catch-up when unblocked.
  - Idempotency: per-instance per-campaign `generation` executes at most once.
  - `generation` increments only on `action/payload` change.
  - Lifecycle: manual `close` + optional TTL; `close` stops scheduling but allows in-flight completion.
- Gate / Facts / Probe
  - Gate is check-and-block only (no repair).
  - v0.1 gate facts: `online`, `gateway_reachable`, `version`, `skills_snapshot` with TTLs per `bulk-management.md`.
  - `skills_snapshot` stale is a hard block until `skills.status` probe writes back.
- Events (L2)
  - Append-only structured timeline with export (JSONL primary, CSV secondary).
  - Every event includes `instanceName` + full label snapshot (business + system).
  - Facts snapshots only for blocked/probe events.
  - Uniform redaction (no tokens/secrets; prefer allowlist + hash when in doubt).
  - Default redaction can be `prefix(4)+sha256` for sensitive strings; avoid storing raw secrets.
  - Raw payload/results for analysis go to a separate "artifact" store:
    - Enabled by default in v0.1.
    - Default retention: 30 days.
    - Storage backend: Postgres (v0.1).
    - No explicit size limits enforced in v0.1.

### 2) Remote Skill Pull + Install

Scope boundary: no arbitrary host shell from the control plane.

Important clarification:
- OpenClaw `skills.install` installs **requirements** for an already-present skill (brew/node/go/uv/download installers).
- "Remote skill pull" here means distributing **skill folders** (e.g. `SKILL.md` + files) onto instances so OpenClaw can load them.

v0.1 must-haves:
- Batch actions:
  - Requirements install via `skills.install` (Gateway).
  - Skill config toggle/config via `skills.update` (Gateway).
  - Remote skill folder distribution via a Sidecar-managed action (exact API TBD), implemented without shell scripts.
  - Source of truth: Sidecar only downloads skill bundles via the control plane.
    - Physical delivery may use control-plane issued short-lived download URLs (redirect / pre-signed URL) backed by object storage + CDN for cross-region performance.
- Post-install fact correctness:
  - Any successful install/update invalidates `skills_snapshot`.
  - Gate must wait for `skills.status` probe writeback before proceeding with skill-dependent work.
- Auditability:
  - Emit events for install/update start/finish and probe refresh.
- Directory isolation:
  - Sidecar installs skill folders under `~/.openclaw-fleet/skills` (not `~/.openclaw/skills`).
  - Sidecar uses `config.patch` to add `~/.openclaw-fleet/skills` into OpenClaw `skills.load.extraDirs`.
- Bundle format: `tar.gz` (v0.1).
- Storage backend: Postgres (v0.1).
- No explicit size limits enforced in v0.1.

Deferred (needs confirmation / later milestones):
- Artifact trust: signing/verification policy.
- Platform compatibility precheck using `os/arch` labels + skill metadata. (explicitly skipped for v0.1)
- Remote skill distribution details:
  - Source type: control-plane stored artifact vs registry integration.
  - Atomic swap implementation details (staging, checksum, swap, cleanup).
    - Preferred implementation is in Sidecar since Gateway is not owned by this project.

### 3) Monitoring / Audit

v0.1 focuses on explainability over full observability stack.

Must-haves:
- Per-campaign timeline + per-instance timeline derived from events.
- Export with 90-day rolling retention; manual export before deletion.
- Minimal metrics can be derived from events (success rate, blocked rate, latency) but do not require a full metrics system.

Deferred:
- Centralized log ingestion, full text search, alerting integrations.

### 4) UI and Control Separation (CLI)

Must-haves:
- Control plane exposes stable APIs for Campaign/Selector/Labels/Events (UI is a client).
- CLI uses the same APIs (no privileged backdoor endpoints).

Deferred:
- RBAC and multi-tenant. (Single-tenant in v0.1)

### 5) Execution Feedback (回显)

Must-haves:
- Timeline events sufficient to answer:
  - Which instances were in scope?
  - Which were blocked (and why)?
  - What executed where, when, and what was the result?
- UI/CLI should support "follow" via polling initially; realtime streaming (SSE/WS) is deferred.

Deferred:
- Automated diagnosis / log-based root-cause analysis.

### 6) Align to K8s (Full Lifecycle)

Layering decision:
- Kubernetes lifecycle alignment (operator/reconciler) can exist, but is invisible to the business control plane.
- The business control plane only sees `Instance` with stable identity, labels, and facts.

v0.1 must-haves:
- Stable identity story for k8s-launched instances (prefer StatefulSet + persisted device token in Secret).
- System labels from the K8s layer are surfaced on Instances for selection (e.g. `openclaw.io/os`, `openclaw.io/arch`, `openclaw.io/provider=k8s`).

Deferred:
- Full CRUD lifecycle (create/delete OpenClaw instances) driven from the control plane.
- Cross-provider instance provisioning abstraction.

## Explicit Out-Of-Scope (v0.1)

- Arbitrary host script execution from the control plane.
- One-off "precondition alignment" that mutates instance baseline during Campaign execution.
  - Baseline alignment must be a long-term `Policy`.
- Multi-tenant isolation and RBAC.
