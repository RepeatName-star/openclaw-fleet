# UI v0.2 Ops Console Design

**Date:** 2026-03-19

**Status:** Approved for implementation

## Goal

Upgrade the current Fleet web UI from a resource-oriented MVP into an operator-oriented control console. The new console should make instance identification, task execution, batch operations, skill management, file editing, and audit inspection significantly easier to use in day-to-day operations.

## Product Direction

The current UI is functionally complete for v0.2.0, but it still exposes backend resources too directly:

- instance display is hostname-first and hard to scan
- tasks and events are too raw and not organized around operator workflows
- labels, groups, skills, and bundles are split across pages in ways that force context switching
- memory editing does not expose the real file surface operators care about
- search, pagination, and Chinese-first navigation are incomplete

The v0.2 ops refresh will reorganize the UI around operator jobs instead of backend object types.

## Information Architecture

The sidebar should become Chinese-first and operator-oriented. The target top-level structure is:

- `全局概览`
- `实例`
- `任务与审计`
- `技能管理`
- `分组与标签`
- `批量任务`
- `文件与记忆`

This replaces the current flat navigation of separate `Tasks`, `Events`, `Labels`, `Groups`, `Skill Bundles`, and `Memory/Persona` pages.

## Required Backend Capability Upgrades

This refresh is not a pure frontend exercise. The following backend upgrades are required:

### Instances

Add stable operator-facing display metadata:

- `display_name`: optional remark/alias set by the operator
- `last_seen_ip`: latest IP observed from sidecar traffic or heartbeat metadata

Display logic:

- primary instance name = `coalesce(display_name, name)`
- raw hostname remains stored in `name`
- all UI search and lookup should treat `display_name` as the preferred identity when present

### Tasks

Add operator-facing naming and search fields:

- `task_name`: optional task label provided at creation time

Display logic:

- primary task label = `coalesce(task_name, action)`

Search/filter support:

- `task_name`
- instance display name / hostname
- `action`
- `status`

### Pagination

Promote all major list endpoints from fixed-limit MVP lists to paginated resources:

- `instances`
- `tasks`
- `events`
- `groups`
- `campaigns`

Standard response shape:

```json
{
  "items": [],
  "total": 0,
  "page": 1,
  "page_size": 10
}
```

### Overview

Add a lightweight `/v1/overview` endpoint for summary cards such as:

- total managed instances
- online instances
- total tasks
- pending / leased / done / error tasks
- open campaigns
- skill bundle count

### Files / Memory

Add control-plane mediated file APIs for a fixed safe whitelist:

- `AGENTS.md`
- `SOUL.md`
- `TOOLS.md`
- `IDENTITY.md`
- `USER.md`
- `HEARTBEAT.md`
- `BOOTSTRAP.md`
- `MEMORY.md`

The browser should not talk to OpenClaw Gateway directly. Instead, the control plane should provide a controlled request/response path that asks the sidecar to:

- list available whitelisted files
- read a selected file
- write a selected file

This keeps auth and gateway topology inside Fleet.

## Page-Level UX Targets

### 全局概览

New landing page with summary cards and quick links to active work:

- instance counts
- task counts by status
- campaign counts
- recent activity

### 实例

Primary table columns:

- 备注名
- Hostname
- IP
- 在线状态
- 最近更新时间

Requirements:

- search by remark first, hostname second
- inline edit or quick edit for remark name

### 任务与审计

This becomes a compound page with tabs:

- `任务`
- `事件`

Task tab:

- primary display: `TaskName`
- visible fields: action, instance remark, status, attempts, updated_at
- search by task name, instance name, action, status

Event tab:

- cleaner visual hierarchy
- keep filters
- allow grouping/jumping by task and campaign context
- preserve access to raw event details and artifacts

### 技能管理

Merge the current skill snapshot page and bundle install flow:

- instance selector
- current skills table
- skill count metric
- right-top `安装 Skill` button
- modal for upload + install flow

The separate bundle page can be removed from primary navigation once this is complete.

### 分组与标签

Merge groups and labels into one workflow-oriented page:

- group list and selector editing
- label editing for selected instances
- match preview nearby

The point is to keep targeting metadata in one place.

### 批量任务

Keep Campaigns as the main batch execution page, but improve payload editing:

- action-aware payload modal
- seed modal with default example JSON
- include field hints and notes
- avoid forcing users to edit raw JSON inline in the main form

### 文件与记忆

Replace current memory/persona page with a file editor:

- left rail: file list
- right pane: file content editor
- save action
- instance selector

This page should feel closer to OpenClaw's Files UI, but remain Fleet-controlled.

## Shared UX Rules

- Chinese-first navigation and labels
- consistent pagination controls
- default page size = 10
- page size override in UI
- remove low-contrast white-on-light text issues
- consistent status badges and filter bars

## Delivery Strategy

Implementation should follow three milestones:

1. Backend foundation:
   - schema
   - pagination
   - search
   - overview
   - task naming
   - instance display metadata
   - file APIs
2. UI information architecture refresh:
   - sidebar
   - new pages
   - merged workflows
3. UX polish:
   - modal payload editor
   - pagination widgets
   - contrast fixes
   - Chinese copy cleanup

## Explicit Non-Goals

- No browser-direct gateway integration
- No arbitrary file browsing outside the approved whitelist
- No rollout scheduler redesign in this phase
- No new execution semantics for campaigns beyond UI/API support improvements
