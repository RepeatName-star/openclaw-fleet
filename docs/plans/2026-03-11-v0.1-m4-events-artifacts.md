# Bulk Management v0.1 - Milestone 4 (Events L2 + Artifacts) Implementation Plan

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

**Goal:** Provide auditable execution feedback (L2 structured events) with safe export (JSONL primary, CSV secondary) and a short-retention artifact store for raw payload/results used in diagnosis and offline analysis.

**Architecture:**
- `events` is append-only with 90-day rolling retention. Every event row redundantly stores:
  - `instance_name`
  - `labels_snapshot` (business + system labels at the time)
- `facts_snapshot` is only populated for `target.blocked/*` and `probe.*` events.
- `artifacts` stores raw payload/results for 30 days. Events reference artifacts by `artifact_id`.
- Redaction is centralized: anything risky is stored as `prefix(4)+sha256` in events; raw goes to artifacts.

**Tech Stack:** Fastify, TypeScript, Postgres (pg), Vitest + pg-mem.

**Prereqs:** Milestone 0 schema, Milestone 2 campaigns, Milestone 3 gate/probe (at least basic facts).

import type { Pool } from "pg";

export async function insertArtifact(
  pool: Pool,
  params: { kind: string; content: unknown; sha256?: string; bytes?: number },
): Promise<{ id: string }> {
  const res = await pool.query(
    "insert into artifacts (kind, content, sha256, bytes) values ($1,$2,$3,$4) returning id",
    [params.kind, params.content ?? {}, params.sha256 ?? null, params.bytes ?? null],

}

**Step 3: Implement event emission**

1. In `src/campaigns/reconciler.ts`:
   - When inserting a new `campaign_instances` row: emit `target.added`.
   - When marking `state='removed'`: emit `target.removed`.
   - When setting `state='blocked'`: emit `target.blocked` with `facts_snapshot` populated.
   - When transitioning blocked -> pending: emit `target.unblocked`.
   - When inserting an execution task: emit `exec.queued`.
     - Store raw task payload as an artifact `kind="task.payload"` (30d), reference it by `artifact_id` in the event.
     - Store only redacted preview in event `payload`.

2. In `src/probe/scheduler.ts`:
   - When scheduling a probe task: emit `probe.requested` (facts snapshot included).

3. In `src/routes/tasks-pull.ts`:
   - After leasing a task: emit `exec.started` or `probe.started` (v0.1 minimum requires `exec.started`; probe started is optional).
   - If the leased task is linked to a campaign (find via `campaign_instances.task_id`), include `campaign_id` and `campaign_generation` in event row.

4. In `src/routes/tasks-ack.ts`:
   - On ok: emit `exec.finished` or `probe.finished` with `artifact_id` pointing to raw result.
   - On error: emit `exec.finished` with status error (artifact contains raw error + redacted summary).
