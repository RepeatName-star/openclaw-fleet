# v0.1.1 Hardening Implementation Plan

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

**Goal:** Harden the shipped v0.1 bulk-management and skill-management feature set by fixing unsafe UI/API behavior, removing stale compatibility paths, tightening gate semantics, clarifying memory operations, and refreshing operator docs.

**Architecture:** Keep the existing v0.1 object model intact and make narrowly scoped corrections. Fix the `Campaign` editing flow by loading full campaign detail before editing, add closed-only campaign deletion with browser-safe aliases, reject legacy `group` direct tasks instead of pretending they still map to selector groups, make gate evaluation action-aware so unrelated tasks are not blocked by `skills_snapshot`, and remove the hidden session reset from `memory.replace`. Finish with a full documentation pass so operator docs match runtime behavior.

**Tech Stack:** TypeScript, Fastify, React, pg/Postgres, Redis, Vitest, Testing Library.

## Confirmed Product Decisions

- Editing an existing `Campaign` must not silently clear `payload`, `gate`, or `rollout`.
- Legacy `target_type="group"` direct tasks should be removed from supported behavior.
- `Group` delete should get the same browser-safe fallback pattern used by labels and skill bundles.
- `Campaign` delete is allowed only when the campaign is already `closed`.
- Deleting a `Campaign` removes only the campaign record and live control-plane linkage; historical tasks/events/artifacts remain intact.
- `Gate` should no longer block unrelated actions on `skills_snapshot`.
- `memory.replace` is a Fleet-defined action and should no longer include an implicit session reset. Reset must be an explicit separate task.
- After behavior changes land, operator docs must be refreshed to match reality.
